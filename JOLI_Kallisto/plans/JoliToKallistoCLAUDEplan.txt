  The Core Insight

  You don't need to modify kallisto's C++. The architecture splits cleanly:

  KEEP from kallisto (steps 1-3):    REPLACE with your EM (step 4):
    kallisto bus                       kallisto quant-tcc
    bustools sort                         ↓
    bustools count                     YOUR Python EM
         ↓                            on equivalence classes
    count.mtx + matrix.ec + transcripts.txt

  Kallisto's pseudoalignment gives you equivalence classes (ECs) + counts as a TCC matrix. Your Python EM replaces kallisto quant-tcc by operating on ECs instead of individual reads — same math, much more efficient

  ---
  Implementation Plan

  Phase 1: Single-Sample EC-Based EM (Core)

  Step 1.1 — TCC Data Loader (JOLI_Kallisto/load_tcc.py)

  Read kallisto's output files into Python data structures:
  - count.mtx → sparse TCC matrix → ec_counts[ec_id] = count
  - matrix.ec → EC definitions → ec_transcripts[ec_id] = [tx_id_1, tx_id_2, ...]
  - transcripts.txt → transcript_names[tx_id] = name
  - Optionally read run_info.json for transcript lengths → eff_lens[tx_id]

  Step 1.2 — Weight Computation (JOLI_Kallisto/weights.py)

  Replicate kallisto's weight calculation:
  # For each EC:
  #   For each transcript t in EC:
  #     weight[ec][t] = 1.0 / eff_lens[t]
  # (effective-length normalization)

  Step 1.3 — EM Algorithm on ECs (JOLI_Kallisto/em_algorithm.py)

  A JoliEM class that mirrors your Expec_Max but operates on ECs:

  Initialization:
    theta[t] = 1 / num_transcripts  (uniform)

  For each EM iteration:
    # E-step + M-step combined (like kallisto):
    n[t] = 0  for all t

    For each EC with ec_counts[ec] > 0:
      denom = sum( theta[t] * weight[ec][t]  for t in ec_transcripts[ec] )
      For each t in ec_transcripts[ec]:
        n[t] += ec_counts[ec] * (theta[t] * weight[ec][t]) / denom

    # Normalize:
    theta = n / sum(n)

    # Convergence check:
    if max(|theta_new - theta_old|) < tol: break

  This is mathematically identical to both kallisto's C++ EM and your AT_code MAP EM (with alpha=1), but operates on ECs instead of individual reads.

  Step 1.4 — Output Writer (JOLI_Kallisto/output_writer.py)

  Compute and write results:
  est_counts[t] = theta[t] * total_reads
  tpm[t] = (theta[t] / eff_lens[t]) / sum(theta / eff_lens) * 1e6
  Output: abundance.tsv with columns transcript_name | est_count | tpm

  Step 1.5 — JOLI EM Entry Point (JOLI_Kallisto/main_joli.py)

  Pure Python CLI: takes a sample dir with TCC files, runs JOLI EM, writes abundance.tsv.
  No subprocess calls — clean Python breakpoint-friendly entry point.

  CLI:
    python main_joli.py \
        --sample_dir  <path containing count.mtx, matrix.ec, transcripts.txt> \
        --output_dir  <where to write abundance.tsv> \
        --max_em_rounds 10000 \
        --eff_len_mode  uniform \
        --em_type       plain

  Step 1.6 — Shell Pipeline (JOLI_Kallisto/run_joli_kallisto.sh)  [NEW FILE, do not edit run_lr_kallisto.sh]

  Shell script that:
    1. Reads CONFIG section (binary paths, index, sample list)
    2. For each sample: checks bustools cache
         cache hit  -> skip steps 1-3
         cache miss -> run kallisto bus + bustools sort + bustools count
                       save outputs to <reads_dir>/kallisto_output/<sample_stem>/
    3. Calls python main_joli.py on the TCC cache dir
    4. Saves results to timestamped output folder

  THIS IS WHERE THE CACHE CHECK LIVES (not in main_joli.py).

  SLURM submission: submit_joli_pipeline.sh (edit existing) -> submits run_joli_kallisto.sh

  Do NOT modify: run_lr_kallisto.sh, submit_lr_kallisto.sh (legacy pipeline, kept intact).

  ---
  Why This Design

  1. Reuse kallisto's pseudoalignment — the hard part (k-mer indexing, EC construction) stays in optimized C++
  2. EC-based EM is orders of magnitude faster than per-read — a dataset with 1M reads might have only 50K ECs
  3. Clean separation — your Python EM is independent of kallisto's C++, easy to iterate on
  4. Same math — EC-based EM is mathematically equivalent to per-read EM (ECs just group identical reads)
  5. Incremental — Phase 1 works standalone; Phases 2-4 add your unique contributions (Dirichlet GD, multi-sample, VI)
